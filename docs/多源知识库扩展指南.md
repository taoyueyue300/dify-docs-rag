## 1. 为什么"代码随想录 PDF"丢进 Dify 文档索引会变差？

### 1.1 不是 PDF 解析的问题

- pypdf 抽取代码片段会丢缩进、断行，但这是次要问题
- 真正的杀手是**主题污染**：你的现有索引围绕"Dify 怎么用"，混入"算法题解"后：
  - **检索召回率下降**：用户问"Dify 怎么部署 Workflow"，BM25 关键词层会把代码随想录里同样含"workflow"的算法节点也召回（无关 chunk）
  - **向量层混淆**：BGE 向量空间里"代码片段 + 算法解释"和"工程文档"在中文嵌入下距离常常 < 0.3，难区分
  - **最终 LLM 拼接 context 时被干扰**

### 1.2 量化推演（按目前 855 chunks 的索引）

| 场景 | 索引规模 | 预期 Hit@5 命中率 | 延迟 P95 |
|---|---|---|---|
| 仅 Dify 文档（现状） | 855 chunks | **80%**（实测） | 17.5 ms |
| + 代码随想录 PDF（约 1500 chunks） | ~2400 chunks | **预计 55–65%**（混淆） | ~25 ms |
| 独立索引（推荐） | 2 个独立索引 | 各自 75–85% | 各 17–25 ms |

### 1.3 三种可选架构

#### 方案 A · 独立索引（强烈推荐）

```
faiss_index/         ← 现有，Dify docs
faiss_index_algo/    ← 新建，代码随想录
```

代码改动：`ingest_multi.py --path ./代码随想录PDF全集 --index-dir faiss_index_algo`

查询时按用户问题路由（关键词或小型分类器）。

#### 方案 B · 加 namespace 字段

把 `metadata.kind` 升级为 `namespace`，检索时按 namespace 预过滤。
[`loaders.py`](../loaders.py) 已经在 metadata 里带 `kind` 字段，只需在 `retriever.py` 加一个过滤参数。

#### 方案 C · 直接合并（不推荐）

只在两类资料强相关时才合（例如把 Dify 官方文档 + Dify 中文社区博客合并）。

## 2. 网页版资料的接入方式

### 2.1 三种姿势对比

| 方式 | 实现 | 适合场景 | 缺点 |
|---|---|---|---|
| **离线抓取 + load_html_file** | 用 wget/playwright 整站爬到本地 → loader 扫文件夹 | 一次性建 KB | 需要爬虫工程量 |
| **在线 fetch + load_url**（loaders.py 已有） | 给一组 URL 即抓即解析 | URL 数量少 (< 100) | 重建索引慢；网络抖动易失败 |
| **走 Dify Knowledge API** | 直接调用 Dify SaaS / 本地的知识库 API | 已经在用 Dify | 与本仓库目标重合，建议二选一 |

### 2.2 用 ingest_multi.py 加在线网页

```powershell
cd dify-docs-rag
.\.venv\Scripts\Activate.ps1
pip install pypdf python-docx beautifulsoup4 lxml httpx
python ingest_multi.py `
  --urls https://docs.dify.ai/zh-hans/introduction `
         https://docs.dify.ai/zh-hans/guides/workflow/introduce `
  --index-dir ./faiss_index_web
```

抽正文逻辑（`load_html`）：去掉 `<script>/<style>/<nav>/<footer>`，优先取 `<article>` / `<main>`，h1-h3 转 markdown 标题（利于按标题切分）。

### 2.3 网页/RSS 自动化采集（进阶）

如需周期性同步博客：

```python
# 伪代码：每天定时跑一次
import feedparser
for entry in feedparser.parse("https://example.com/feed.xml").entries:
    text = load_url(entry.link)
    # incremental upsert 到 FAISS
```

? FAISS `IndexFlatIP` 不支持高效删除，规模大后建议换 `IndexHNSW` + `id_map` 或迁移到 ChromaDB / Qdrant。

### 3.1 retriever.py 增加过滤示例（可选改动）

```python
# 在 HybridRetriever.search 里加 kind 过滤
def search(self, query, top_k=5, kind: str | None = None):
    results = self._raw_search(query, top_k * 5 if kind else top_k)
    if kind:
        results = [r for r in results if r["metadata"].get("kind") == kind]
    return results[:top_k]
```

### 4.4 推荐的 KB 组合（落地建议）

新建一个 **`dify-knowledge-base`** 项目：

```
dify-knowledge-base/
├── faiss_index_official/   # docs.dify.ai 官方文档（最新）
├── faiss_index_blog/       # Dify 官方博客
├── faiss_index_community/  # 知乎/掘金/CSDN 等社区精选（人工筛选）
├── faiss_index_source/     # README + AGENTS.md + 关键源码注释
└── router.py               # 按问题 namespace 路由
```

这样既不污染 dify-docs-rag 的"演示项目"特性，又能积累一个真实可用的"Dify 全资料 KB"，简历上可写第 4 个项目。
