## 测试目标

补齐 RAG 项目的"性能侧"数据，配合已有的"效果侧"实验报告，形成**效果 + 性能**双维度证据。

## 测试环境

- Windows 10 · Python 3.11 · 本地 CPU
- 索引规模：**855 chunks**（chunk_size=800，55 篇 Dify 官方文档）
- Embedding 模型：`BAAI/bge-small-zh-v1.5`
- 检索策略：FAISS 向量 + BM25 加权融合（无 Reranker，已验证为最优配置）
- 测试样本：10 个真实问题 × 15 轮 = 150 次检索

## 关键结果

### 检索延迟分布

| 指标 | 值 |
|---|---:|
| 模型/索引初始化 | 11.5 s |
| 索引规模 | 855 chunks |
| **冷启动首次检索** | 126 ms |
| 稳态 P50 | **14.2 ms** |
| 稳态 P90 | 16.5 ms |
| 稳态 P95 | **17.5 ms** |
| 稳态 P99 | 18.4 ms |
| 稳态 Max | 21.0 ms |
| 平均吞吐 | **69.6 QPS** |

### 与效果数据交叉对照

| 维度 | 数值 | 结论 |
|---|---|---|
| 命中率 (Hit@5) | **80%** ([experiment_report.md](../experiment_report.md)) | 端到端检索有效 |

| 端到端延迟（含 LLM） | ≈ LLM_latency + 18 ms | 检索仅占 < 5% |

## 关键设计选择（面试讲法）

1. **不用 Reranker**：在 855 chunks 的中文技术文档场景下，BGE-reranker-base **降低了 30%** 命中率（详见 experiment_report.md 实验3），改用加权融合：向量 0.6 + BM25 0.4。
2. **chunk_size=800**：实验对比 300/500/800，800 命中率最高（70%→80%），原因是技术文档段落语义完整性优先于切分粒度。
3. **CPU 也能跑 70 QPS**：bge-small-zh 仅 33M 参数，单 query 向量化 < 10 ms，FAISS IVF 索引内存检索 < 2 ms。
